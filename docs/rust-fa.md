# راهنمای نسخهٔ Rust — لینوکس / مک / ویندوز

> English: [Rust application guide](rust-en.md)

نسخهٔ Rust (`bale-vpn-rust/`) یک باینری تک‌فایلیِ **کلاینت یا سرور VPN** است که یک رابط HTTP داخلی (و در صورت دلخواه، یک پنجرهٔ دسکتاپ بومی) دارد. جایگزین نسخهٔ Node است — همان قالب پیکربندی روی دیسک، با امکانات بیشتر و بدون نیاز به Node.

- باینری برای لینوکس، مک و ویندوز.
- دو نقش که در اولین اجرا از طریق UI انتخاب می‌شوند یا با ساب‌فرمان CLI ثابت می‌شوند:
  - **server** — تماس‌های ورودی بله از مخاطبینِ مجاز را می‌پذیرد و ترافیک IP هر تماس‌گیرنده را به اینترنت آزاد منتقل می‌کند.
  - **client** — تماسی به یک peer پیکربندی‌شده می‌گذارد، یک شنونده محلی SOCKS5 (و اختیاراً یک TUN سیستمی) روی تونل حاصل اجرا می‌کند.
- دو راهبرد فوروارد سمتِ سرور:
  - **kernel TUN** — هسته فوروارد IP و NAT را انجام می‌دهد (`iptables` MASQUERADE روی لینوکس، اَنکر `pf` روی مک). بالاترین پهنای‌باند. به یک مرحله تنظیم یک‌بارهٔ root نیاز دارد.
  - **userspace NAT** — بدون دسترسی ویژه اجرا می‌شود. ماشین حالت TCP / UDP در فضای کاربری داخل پروسه.
- یک **رابط وب محلی** روی `http://localhost:3001` سطح پیکربندی است — ورود، انتخاب حالت، مدیریت فهرست مجاز، درخواست‌های در انتظار، نمایش کلاینت‌های متصل، انتخاب peer در حالت کلاینت.
- در بیلدهای دسکتاپ، همان رابط وب به‌صورت خودکار در یک **پنجرهٔ بومی** هم باز می‌شود (با `--headless` می‌توان رد کرد).

## نگاهی کلی به حالت‌ها

| حالت | نقش | سطح دسترسی | پهنای‌باند |
|---|---|---|---|
| `server --nat-mode userspace` (پیش‌فرض) | سرور | بدون دسترسی ویژه | پایین‌تر از حالت هسته اما کاملاً قابل اعتماد (SACK، RACK، TLP، PRR، Window Scaling، Timestamps) |
| `server --nat-mode kernel` | سرور | تنظیم یک‌بارهٔ root (روی لینوکس `setcap` + `iptables`؛ روی مک با root اجرا می‌شود) | بالاترین — دستگاه TUN هسته + NAT بومی |
| `client` | کلاینت | بدون دسترسی ویژه | پراکسی SOCKS5 روی `127.0.0.1:1080` از طریق کریر RTP تماس (توسط QUIC) |
| `client --client-tun` | کلاینت | `CAP_NET_ADMIN` (لینوکس) یا root (مک) | علاوه بر بالا، یک TUN سیستمی روی `10.8.0.2/24` که اجازه می‌دهد کل میزبان ترافیک را از تونل عبور دهد |

پیش‌فرض NAT روی `userspace` است (در نسخهٔ Node پیش‌فرض `kernel` بود). اگر بیشترین پهنای‌باند می‌خواهید و تنظیم یک‌بارهٔ پایین را انجام داده‌اید، `kernel` را به‌صراحت انتخاب کنید.

## شروع سریع

### ساخت از سورس

```bash
git clone <repo>
cd BaleVPN
make build                                         # = cargo build --release
./bale-vpn-rust/target/release/bale-vpn            # پنجره را باز می‌کند
```

ترابری اکنون کاملاً Rust است (`webrtc-rs` به‌همراه `livekit-api`/`livekit-protocol` از crates.io) — دیگر هیچ کرت vendor‌شده و هیچ مرحلهٔ پچی وجود ندارد، پس `make build` صرفاً `cargo build --release` است (یا مستقیماً `cd bale-vpn-rust && cargo build --release` را اجرا کنید).

برای استقرارهای headless / سرور (باینری کوچک‌تر، بدون وابستگی tao/wry):

```bash
make build-headless
./bale-vpn-rust/target/release/bale-vpn server --nat-mode userspace
```

### اجرای باینری منتشرشده

باینری ساخته‌شده را از صفحهٔ [Releases](../../../releases) مخزن دانلود کنید. هر نسخه برای هر سیستم‌عامل دو نوع دارد:

- `bale-vpn-gui-<os>-<arch>` — باینری پنجره‌دار دسکتاپ (با `--headless` می‌توان پنجره را رد کرد).
- `bale-vpn-headless-<os>-<arch>` — فقط HTTP، کوچک‌تر، بدون وابستگی به webview سیستمی. مناسب استقرار سرور.

```bash
chmod +x bale-vpn-*-*
./bale-vpn-gui-macos-aarch64                       # پنجره را باز می‌کند، پورت پیش‌فرض ۳۰۰۱
./bale-vpn-headless-linux-x86_64 server            # حالت سرور، NAT فضای کاربری
./bale-vpn-headless-linux-x86_64 client            # حالت کلاینت، فقط SOCKS5
```

### آرگومان‌های خط فرمان

CLI از ساب‌فرمان استفاده می‌کند تا پرچم‌های مخصوصِ هر حالت فقط زیر شاخهٔ درست ظاهر شوند:

```
bale-vpn                                         # بدون حالت → UI انتخاب‌گر را نشان می‌دهد
bale-vpn server                                  # سرور، NAT فضای کاربری پیش‌فرض
bale-vpn server --nat-mode kernel                # سرور، TUN هسته
bale-vpn client                                  # کلاینت، فقط SOCKS5
bale-vpn client --client-tun                     # کلاینت، SOCKS5 + TUN سیستمی
bale-vpn --port 8080 server --nat-mode kernel    # ترکیب آرگومان‌های سطح بالا و مخصوص حالت
bale-vpn --headless                              # بیلد GUI، پنجره را رد کن
```

| آرگومان | حوزه | پیش‌فرض | توضیح |
|---|---|---|---|
| `--port <int>` | سطح بالا | `3001` | پورت رابط HTTP. به `127.0.0.1` بسته می‌شود. |
| `--headless` | سطح بالا | خاموش | فقط بیلدهای GUI: پنجرهٔ بومی را باز نکن. |
| `--config-dir <path>` | سطح بالا | کنار باینری | محل ذخیرهٔ `.bale-vpn_config.json`. |
| `server --nat-mode kernel\|userspace` | server | `userspace` | راهبرد فوروارد سمت سرور را انتخاب می‌کند. حالت `kernel` به تنظیم یک‌بارهٔ پایین نیاز دارد. |
| `client --client-tun` | client | خاموش | یک TUN محلی روی `10.8.0.2/24` در کنار SOCKS5 باز می‌کند تا کل میزبان از تونل عبور کند. |

اگر ساب‌فرمان مدِ خاصی پاس نشود، دیمن در ابتدای کار پارک می‌شود و UI یک **انتخاب‌گر حالت** (Client / Server) نشان می‌دهد. پس از انتخاب، انتخاب در فایل پیکربندی ذخیره می‌شود و دفعهٔ بعد انتخاب‌گر نشان داده نمی‌شود. پاس کردن مد از طریق ساب‌فرمان انتخاب‌گر را غیرفعال می‌کند.

رابط وب فقط روی **`127.0.0.1`** گوش می‌دهد — از روی شبکه قابل دسترس نیست. برای یک سرور بدون مانیتور، از فوروارد پورت SSH استفاده کنید:

```bash
ssh -L 3001:127.0.0.1:3001 user@your-server
```

سپس `http://localhost:3001` را در مرورگر محلی باز کنید.

## تنظیم یک‌بارهٔ kernel-TUN (حالت سرور)

### لینوکس

```bash
# ۱. اجازهٔ مدیریت TUN بدون root به باینری بدهید.
sudo setcap cap_net_admin+eip ./target/release/bale-vpn

# ۲. فوروارد IPv4 را فعال کنید (و در ریبوت‌ها هم باقی بماند).
sudo sysctl -w net.ipv4.ip_forward=1
echo 'net.ipv4.ip_forward = 1' | sudo tee /etc/sysctl.d/99-bale-vpn.conf

# ۳. ترافیک خروجی از pool تونل را به اینترفیس واقعی NAT کنید. سرور
#    Rust از یک /24 جداگانه برای هر کلاینت در 10.8.0.0/16 استفاده
#    می‌کند، پس قاعدهٔ MASQUERADE کل /16 را پوشش می‌دهد.
sudo iptables -t nat -A POSTROUTING -s 10.8.0.0/16 -j MASQUERADE
```

سپس اجرا کنید:

```bash
./target/release/bale-vpn server --nat-mode kernel
```

### مک

مک از `pf` (Packet Filter) به‌جای iptables استفاده می‌کند. حالت kernel-TUN با root اجرا می‌شود:

```bash
# فعال‌سازی فوروارد IP (پایدار در ریبوت):
sudo sysctl -w net.inet.ip.forwarding=1
echo 'net.inet.ip.forwarding=1' | sudo tee -a /etc/sysctl.conf

# اینترفیس WAN را پیدا کنید (معمولاً en0 روی WiFi):
route get default | grep interface

# اَنکر pf را بسازید. <wan> را با اینترفیس خود جایگزین کنید (مثلاً en0):
cat <<'EOF' | sudo tee /etc/pf.anchors/balevpn
nat on <wan> from 10.8.0.0/16 to any -> (<wan>)
pass on utun0
pass on utun1
EOF

# آن را به پیکربندی اصلی pf متصل کنید:
echo 'nat-anchor "balevpn"'                                | sudo tee -a /etc/pf.conf
echo 'rdr-anchor "balevpn"'                                | sudo tee -a /etc/pf.conf
echo 'anchor "balevpn"'                                    | sudo tee -a /etc/pf.conf
echo 'load anchor "balevpn" from "/etc/pf.anchors/balevpn"' | sudo tee -a /etc/pf.conf

# pf را بارگذاری مجدد + فعال کنید:
sudo pfctl -f /etc/pf.conf
sudo pfctl -e

# با root اجرا کنید (utun به سوکت کنترل PF_SYSTEM نیاز دارد):
sudo ./target/release/bale-vpn server --nat-mode kernel
```

مک نام‌گذاری `utun*` را اجباری می‌کند — راهنمای `bale<K>` دیمن نادیده گرفته می‌شود؛ نام واقعی را در لاگ `utun: opened utunN ...` ببینید.

### ویندوز

TUN هسته روی ویندوز هنوز پشتیبانی نمی‌شود. از NAT فضای کاربری استفاده کنید.

```cmd
bale-vpn.exe server --nat-mode userspace
```

### userspace NAT (روی هر سیستم‌عامل، بدون تنظیم)

```bash
./target/release/bale-vpn server --nat-mode userspace
```

### کارهایی که سرور در زمان راه‌اندازی انجام می‌دهد

۱. `access_token` بله و فهرست‌های مجاز / مسدود / max-clients از `<binary-dir>/.bale-vpn_config.json` بارگذاری می‌شوند.

۲. در حالت kernel-TUN: یک TUN آزمایشی برای پیش‌بررسی دسترسی‌ها باز می‌شود تا خطاهای دسترسی در راه‌اندازی آشکار شوند. برای هر تماس پذیرفته‌شده، یک slot K (۰ تا ۲۵۳) رزرو می‌شود و `bale<K>` روی `10.8.K.1/24` (لینوکس) یا یک `utunN` تازه (مک) باز می‌شود. قاعدهٔ MASQUERADE / pf بالا کل `/16` را پوشش می‌دهد.

۳. رابط وب روی پورت انتخابی شروع به کار می‌کند.

۴. به WebSocket سیگنالینگ بله متصل می‌شود و منتظر تماس‌های ورودی می‌ماند.

وقتی یک کلاینت متصل می‌شود، آدرس `10.8.K.2/24` می‌گیرد و ترافیکش وارد تونل می‌شود. در حالت هسته تا ۲۵۴ کلاینت هم‌زمان (هر کدام یک slot /24)؛ در حالت userspace محدود به Max-clients (پیش‌فرض ۵، حداکثر ۲۵۳) است.

### محدودیت‌ها

- فقط IPv4. بسته‌های IPv6 از کلاینت دور انداخته می‌شوند (کلاینت با ICMPv6 Destination Unreachable به‌سرعت به IPv4 برمی‌گردد).
- حالت (`server` در مقابل `client`) و حالت NAT (`kernel` در مقابل `userspace`) در زمان راه‌اندازی ثابت می‌شوند. برای تغییر، باید مجدد اجرا شود.

## رابط وب

UI روی `http://localhost:<port>` در دسترس است. در بیلدهای GUI به‌صورت خودکار در یک پنجرهٔ بومی هم باز می‌شود.

۱. **انتخاب حالت** — در اولین اجرا و وقتی مدِ خاصی از CLI پین نشده، نمایش داده می‌شود. **Client** یا **Server** را انتخاب کنید. انتخاب ذخیره می‌شود.

۲. **ورود** — شماره موبایل ← کد پیامکی، یا چسباندن مستقیم یک کوکی JWT `access_token` از `web.bale.ai`. توکن سمت سرور در `<binary-dir>/.bale-vpn_config.json` (mode 0600) ذخیره می‌شود؛ مرورگر فقط یک بولی `tokenSet` می‌بیند، نه خود JWT.

   <p align="center"><img src="screens/07-node-login.png" alt="صفحهٔ ورود" width="640"></p>

۳. **حالت سرور** — پس از ورود، UI این موارد را نشان می‌دهد:
   - **کلاینت‌های متصل** با سرعت لحظه‌ای، مجموع بایت‌ها، آپ‌تایم هر کلاینت و دکمهٔ قطع.
   - **درخواست‌های در انتظار** — ردیف‌های زرد برای تماس‌های ورودی از مخاطبینی که در فهرست مجاز نیستند. هر ردیف دکمه‌های **یک‌بار قبول / همیشه قبول / رد** دارد. دکمهٔ Reject چسبنده است — تماس‌گیرنده را به فهرست مسدود اضافه می‌کند (با Unblock در فهرست **تماس‌گیرنده‌های مسدود** قابل بازگشت).

     <p align="center"><img src="screens/08-server-client-pending.png" alt="درخواست در انتظار سرور" width="640"></p>

   - فهرست‌های **تماس‌گیرنده‌های مجاز** و **تماس‌گیرنده‌های مسدود** با دکمه‌های Remove / Unblock در هر ردیف.

     <p align="center"><img src="screens/09-server-client-connected.png" alt="کلاینت متصل به سرور" width="640"></p>

   - تنظیم **حداکثر کلاینت** (۱ تا ۲۵۳) برای سقف اتصالات هم‌زمان.

۴. **حالت کلاینت** — پس از ورود، UI این موارد را نشان می‌دهد:
   - انتخاب‌گر **peer سرور** (drop-down از مخاطبین بله + جست‌وجوی "find peer by phone").
   - **پورت SOCKS5** (پیش‌فرض ۱۰۸۰) — شنوندهٔ محلی.
   - دکمهٔ **Connect / Disconnect**. دیمن در راه‌اندازی **بی‌کار** شروع می‌شود — حتی اگر peer ذخیره‌شده‌ای داشته باشد، خودکار وصل نمی‌شود. برای برقراری تماس، **Connect** را بزنید؛ برای پایان، **Disconnect**.
   - وضعیت لحظه‌ای: "در حال اتصال…" ← شمارنده‌های پهنای‌باند پس از برقراری اتاق.

   نشان‌گر WebSocket پیرو موتور قاعده است، نه UI:
   - **connected** — دست‌دهی WS تمام شده و آمادهٔ RPC است (تماس فعال نیست).
   - **paused (in call)** — موتور قاعده WS را عمداً پایین آورده چون یک تماس کلاینت در جریان است. کانال push وسط تماس نیاز نیست؛ پس از پایان تماس خودکار دوباره برمی‌گردد. به‌رنگ سبز آرام نشان داده می‌شود، نه قرمز.
   - **connecting…** — فقط وقتی حلقهٔ اجرا واقعاً در حال تلاش برای دست‌دهی / اتصال مجدد است، نه وقتی نیت ثبت شده ولی موتور عمداً WS را پایین نگه داشته.
   - **disconnected** — توکن وجود ندارد، یا کاربر خروج کرده است.

   <p align="center"><img src="screens/10-client-peer-selection.png" alt="کلاینت: انتخاب peer" width="640"></p>
   <p align="center"><img src="screens/11-client-connected.png" alt="کلاینت: متصل" width="640"></p>

۵. **خروج** توکن ذخیره‌شده را پاک می‌کند و تونل فعال را قطع می‌کند؛ حالت هم پاک می‌شود مگر اینکه از طریق CLI پین شده باشد.

## کنترل پذیرش سرور

هر تماس ورودی از مخاطبی که در فهرست مجاز نیست در صف **در انتظار** قرار می‌گیرد.

- **یک‌بار قبول** فقط همین تماس را می‌پذیرد و تماس‌گیرنده را پایدار نمی‌کند. تماس بعدی دوباره در صف انتظار قرار می‌گیرد.
- **همیشه قبول** تماس‌گیرنده را به فهرست مجاز اضافه می‌کند (زیر کلید `admission` در فایل پیکربندی ذخیره می‌شود). تماس‌های بعدی به‌صورت خودکار پذیرفته می‌شوند.
- **رد** چسبنده است — `DiscardCall` می‌فرستد تا تونل تماس‌گیرنده بلافاصله قطع شود *و* شناسهٔ تماس‌گیرنده را به **فهرست مسدود** اضافه می‌کند. تماس‌های بعدی از این شناسه بی‌صدا رد می‌شوند (بدون اعلان و بدون درخواست انتظار). برای لغو، از فهرست **تماس‌گیرنده‌های مسدود** دکمهٔ Unblock همان ردیف را بزنید.
- در صورت بی‌پاسخ ماندن، درخواست‌های در انتظار بعد از ۶۰ ثانیه به‌طور خودکار رد می‌شوند. این تایم‌اوت تماس‌گیرنده را مسدود **نمی‌کند** — فقط دکمهٔ صریح Reject این کار را می‌کند.

فهرست مجاز و فهرست مسدود دو طرف منحصربه‌فرد هستند — هر Allow / Reject صریح، تماس‌گیرنده را بین این دو جابه‌جا می‌کند.

تنظیم **حداکثر کلاینت** (۱ تا ۲۵۳، پیش‌فرض ۵) سقف تماس‌های هم‌زمان را تعیین می‌کند. تماس‌های ورودی بیش از این سقف بدون مسدود کردن حذف می‌شوند؛ تماس‌گیرنده می‌تواند بعداً وقتی جا باز شد دوباره تلاش کند.

هر سه (admission، blacklist، max-clients) در هر تغییر روی دیسک ذخیره می‌شوند و در راه‌اندازی دیمن دوباره بارگذاری می‌گردند.

## جزئیات حالت کلاینت

در حالت کلاینت، دیمن یک تماس به peer سرورِ پیکربندی‌شده برقرار می‌کند. کریر رسانه‌ای RTP حاصل (webrtc-rs)، QUIC را حمل می‌کند و شنوندهٔ محلی SOCKS5 هر اتصال پذیرفته‌شده را از طریق QUIC به سرور می‌فرستد، که در آن‌جا به اینترنت واقعی متصل می‌شود.

- **فقط SOCKS5** (`bale-vpn client`): شنوندهٔ SOCKS5 به‌طور پیش‌فرض روی `127.0.0.1:1080` گوش می‌دهد. اپ‌های جداگانه (مرورگر، curl و …) را برای استفاده از این پراکسی پیکربندی کنید.
- **SOCKS5 + TUN سیستمی** (`bale-vpn client --client-tun`): علاوه بر SOCKS5، یک دستگاه TUN روی `10.8.0.2/24` باز می‌شود. مسیریابی به عهدهٔ شما است — معمولاً:

  ```bash
  # لینوکس:
  sudo ip route add default dev bale-c0
  # مک (از نام utunN که دیمن لاگ کرده استفاده کنید):
  sudo route add default -interface utun5
  ```

  برای لغو، `add` را با `delete` جایگزین کنید. بدون این مسیرها، دستگاه TUN وجود دارد ولی سیستم ترافیک را از آن نمی‌فرستد.

اگر سمتِ LK قطع شود کلاینت دوباره وصل می‌شود، اما قطعِ دائمی (peer سرور اتاق را ترک کرده) فوراً تونل را برمی‌چیند و UI به prompt Activate برمی‌گردد، بدون انتظار ۳۰+ ثانیه‌ای که اگر فقط به رویداد WS بله تکیه می‌کردیم می‌داشتیم.

## احراز هویت

دو راه برای وارد کردن JWT `access_token`:

۱. **OTP موبایل از UI** — شماره را وارد کنید، کد پیامکی را تایپ کنید؛ باینری کوکی را از مسیر استاندارد `web.bale.ai/set-cookie/?jwt=…` می‌گیرد و در `<binary-dir>/.bale-vpn_config.json` (mode 0600) ذخیره می‌کند. روش توصیه‌شده.

۲. **چسباندن مستقیم توکن** — کوکی `access_token` را از یک نشست لاگین‌شدهٔ `web.bale.ai` در Chrome (DevTools → Application → Cookies) کپی کنید و در تکست‌اریا روی UI بچسبانید.

کد بستن WebSocket `4401` یعنی توکن منقضی شده؛ دوباره وارد شوید.

## فایل پیکربندی

تنها یک فایل JSON در `<binary-dir>/.bale-vpn_config.json`، با mode 0600 و نوشتن atomic-rename. شکل فیلدها با نسخهٔ Node یکی است، پس می‌توان یک نصب موجود را با باینری Rust جایگزین کرد بدون نیاز به ورود مجدد یا از دست رفتن وضعیت سمت سرور.

```json
{
  "token":       "<JWT access_token بله>",
  "mode":        "server",
  "natMode":     "userspace",
  "port":        3001,
  "peerId":      "1234567890",
  "socks5Port":  1080,
  "admission":   ["9876543210"],
  "blacklist":   [],
  "maxClients":  5
}
```

`mode` تا زمانی که کاربر از UI انتخاب کند (یا `--mode` از CLI پاس شود) `null` است.

## بیلد GUI

`cargo build --release` روی هر سیستم‌عامل یک باینری پنجره‌دار تولید می‌کند. روی مک پنجرهٔ دسکتاپ به‌طور خودکار باز می‌شود با:

- منوی Edit متصل به میان‌برهای استاندارد (Cmd+C / Cmd+V / Cmd+X / Cmd+A / Cmd+Z / Cmd+⇧Z).
- آیتم‌های Hide / Quit در منوی برنامه.
- راست‌کلیک ← **Inspect Element** برای webview (devtools فعال است).
- Ctrl+C در ترمینال صدا کننده، پنجره را تمیز می‌بندد (signal handler به event loop wry وصل است، دیمن را تا ۳ ثانیه drain می‌کند قبل از خروج).

برای رد کردن پنجره در یک بیلد GUI، `--headless` را اضافه کنید. برای استقرارهای سرور با `--no-default-features` بیلد کنید تا وابستگی‌های `tao` / `wry` به‌کلی حذف شوند.

## امنیت و رمزنگاری

اتصال داده‌ای بین کلاینت و سرور با DTLS رمزنگاری می‌شود، پس ترافیک از دید ناظرهای منفعل شبکه پنهان است. **اما** سرورهای LiveKit بله نقش SFU را بازی می‌کنند و به محتوای رمزگشایی‌شده دسترسی دارند — می‌توانند مقصدها و هر بارِ اپلیکیشنی غیررمزنگاری‌شده را ببینند. در سطح برنامه از TLS (HTTPS، DNS رمزنگاری‌شده و …) استفاده کنید و این تونل را مثل VPN شرکتی‌ای ببینید که گرداننده‌اش را کاملاً نمی‌شناسید.
