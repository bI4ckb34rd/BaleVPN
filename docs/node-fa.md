# راهنمای نسخهٔ Node — لینوکس / مک

> English: [Node.js application guide](node-en.md)

نسخهٔ Node (`bale-vpn-node/`) یک **سرور VPN** تک‌فایلی است. تماس‌های ورودی بله را از مخاطبینِ مجاز می‌پذیرد و ترافیک IP هر تماس‌گیرنده را به اینترنت آزاد منتقل می‌کند.

- باینری برای لینوکس و مک ساخته می‌شود.
- دو راهبرد فوروارد که در زمان اجرا انتخاب می‌شوند:
  - **kernel TUN** — هسته فوروارد IP و NAT را انجام می‌دهد (`iptables` MASQUERADE روی لینوکس، اَنکر `pf` روی مک). بالاترین پهنای‌باند. به یک مرحله تنظیم یک‌بارهٔ root نیاز دارد.
  - **userspace NAT** — بدون دسترسی ویژه اجرا می‌شود. ماشین حالت TCP / UDP در فضای کاربری داخل پروسه.
- یک **رابط وب محلی** روی `http://localhost:3001` تنها سطح پیکربندی است — ورود، مدیریت فهرست مجاز، درخواست‌های در انتظار، نمایش کلاینت‌های متصل.

> نسخهٔ Node در حال حاضر فقط **حالت سرور** را پشتیبانی می‌کند. برای سمت کلاینت از [اپ اندرویدی](android-fa.md) استفاده کنید.

## نگاهی کلی به حالت‌ها

| حالت | سطح دسترسی | پهنای‌باند |
|---|---|---|
| `--nat-mode kernel` | تنظیم یک‌بارهٔ root (روی لینوکس `setcap` + `iptables`؛ روی مک با root اجرا می‌شود) | بالاترین — دستگاه TUN هسته + NAT بومی |
| `--nat-mode userspace` | بدون دسترسی ویژه | پایین‌تر از حالت هسته اما کاملاً قابل اعتماد (SACK، RACK، TLP، PRR، Window Scaling، Timestamps) |

پیش‌فرض `kernel` است.

## شروع سریع

### اجرای باینری منتشرشده

آخرین فایل `balevpn-<version>-<platform>` را از صفحهٔ [Releases](../../../releases) مخزن دانلود کنید.

```bash
chmod +x balevpn-<version>-{linux,macos}-*
./balevpn-...                             # پورت پیش‌فرض ۳۰۰۱، حالت پیش‌فرض
./balevpn-... 8080                        # پورت UI سفارشی
./balevpn-... --nat-mode=userspace        # حالت userspace را اجباری کن
```

### آرگومان‌های خط فرمان

| آرگومان | پیش‌فرض | توضیح |
|---|---|---|
| `<integer>` (موقعیتی) | `3001` | پورت HTTP رابط مدیریت. هر عدد ساده‌ای به‌عنوان پورت در نظر گرفته می‌شود. |
| `--nat-mode kernel\|userspace` | `kernel` | روش فوروارد سرور را انتخاب می‌کند. `kernel` به تنظیم یک‌بارهٔ پایین نیاز دارد؛ `userspace` بدون دسترسی ویژه کار می‌کند. حالت در زمان شروع ثابت می‌شود. |

اگر `--nat-mode=kernel` انتخاب شود ولی دسترسی‌های هسته یا قاعدهٔ MASQUERADE `iptables` فراهم نباشد، پروسه با یک پیام خطای روشن خارج می‌شود (به‌جای ادامهٔ بی‌سروصدا).

رابط وب فقط روی **`127.0.0.1`** گوش می‌دهد و از روی شبکه قابل دسترس نیست. برای یک سرور بدون مانیتور، از فوروارد پورت محلی SSH استفاده کنید:

```bash
ssh -L 3001:127.0.0.1:3001 user@your-server
```

سپس `http://localhost:3001` را در مرورگر محلی باز کنید. تونل فوروارد باید در زمان استفاده از UI باز بماند.

## تنظیم یک‌بارهٔ kernel-TUN

### لینوکس

```bash
# ۱. اجازهٔ مدیریت TUN بدون root به باینری بدهید.
#    setcap را روی همان فایلی اعمال کنید که اجرا می‌شود — نه روی /usr/bin/node.
sudo setcap cap_net_admin+eip ./balevpn-<version>-linux-x64

# ۲. فوروارد IPv4 را فعال کنید (و در ریبوت‌ها هم باقی بماند).
sudo sysctl -w net.ipv4.ip_forward=1
echo 'net.ipv4.ip_forward = 1' | sudo tee /etc/sysctl.d/99-bale-vpn.conf

# ۳. ترافیک خروجی از سابنت تونل را به اینترفیس واقعی NAT کنید.
sudo iptables -t nat -A POSTROUTING -s 10.8.0.0/24 -j MASQUERADE
```

سپس اجرا کنید:

```bash
./balevpn-<version>-linux-x64
```

### مک

روی مک معادل `setcap` وجود ندارد، پس حالت kernel-TUN با root اجرا می‌شود. قاعدهٔ NAT (anchor `balevpn` در pf) و فوروارد IP در زمان اجرا به‌صورت خودکار بارگذاری می‌شوند؛ اینترفیس WAN با `route -n get default` تشخیص داده می‌شود.

```bash
sudo ./balevpn-<version>-macos-arm64
```

### userspace NAT

هیچ تنظیم خاصی نیاز نیست. فقط کافی است باینری را با `--nat-mode=userspace` اجرا کنید.

```bash
./balevpn-<version>-linux-x64 --nat-mode=userspace
```

### کارهایی که در زمان راه‌اندازی انجام می‌شود

۱. `access_token` ذخیره‌شدهٔ بله (در صورت وجود) از `${RUNTIME_DIR}/.bale-vpn_config.json` بارگذاری می‌شود.

۲. در حالت kernel-TUN: اینترفیس TUN قبلی پاک می‌شود، یکی جدید ساخته می‌شود، آدرس `10.8.0.1/24` تخصیص می‌یابد، فوروارد IPv4 فعال می‌شود و قاعدهٔ NAT پلتفرم (iptables MASQUERADE روی لینوکس، اَنکر pf روی مک) بارگذاری می‌شود.

۳. رابط وب روی پورت انتخابی شروع به کار می‌کند.

۴. به WebSocket سیگنالینگ بله متصل می‌شود و منتظر تماس‌های ورودی می‌ماند.

وقتی یک کلاینت اندرویدی وصل می‌شود، آدرس `10.8.0.2/24` به او داده می‌شود و تمام ترافیکش وارد تونل می‌گردد. تا ۲۵۳ کلاینت می‌توانند هم‌زمان وصل شوند — سرور به‌طور شفاف آدرس مبدأ هر کلاینت را به یک IP متمایز در `10.8.0.0/24` بازنویسی می‌کند تا هسته (یا NAT فضای کاربری) بتواند جریان‌های هم‌زمان را از هم تشخیص دهد.

### محدودیت‌ها

- فقط IPv4. بسته‌های IPv6 از کلاینت دور انداخته می‌شوند (کلاینت اندرویدی با ICMPv6 Destination Unreachable به‌سرعت به IPv4 برمی‌گردد).
- حالت (`kernel` در مقابل `userspace`) در زمان راه‌اندازی ثابت می‌شود. برای تغییر، باید مجدد اجرا شود.

## رابط وب

UI روی `http://localhost:<port>` در دسترس است. این تنها سطح پیکربندی است؛ گزینهٔ پیکربندی از طریق ترمینال وجود ندارد.

۱. **ورود** — شماره موبایل ← کد پیامکی، یا چسباندن مستقیم یک کوکی JWT `access_token` از `web.bale.ai`. توکن سمت سرور در `${RUNTIME_DIR}/.bale-vpn_config.json` (mode 0600) ذخیره می‌شود؛ مرورگر فقط یک بولی `tokenSet` می‌بیند، نه خود JWT.

   <p align="center"><img src="screens/07-node-login.png" alt="صفحه ورود" width="640"></p>

۲. **کلاینت‌های متصل** — سرعت لحظه‌ای، مجموع بایت‌ها، زمان آپ‌تایم هر کلاینت، و دکمهٔ قطع.

۳. **درخواست‌های در انتظار** — ردیف‌های زرد برای تماس‌های ورودی از مخاطبینی که در فهرست مجاز نیستند. هر ردیف دکمه‌های **یک‌بار قبول / همیشه قبول / رد** دارد.

۴. فهرست‌های **تماس‌گیرنده‌های مجاز** و **تماس‌گیرنده‌های مسدود** با دکمه‌های Remove / Unblock در هر ردیف.

## کنترل پذیرش سرور

هر تماس ورودی از مخاطبی که در فهرست مجاز نیست در صف **در انتظار** قرار می‌گیرد.

- **یک‌بار قبول** فقط همین تماس را می‌پذیرد و تماس‌گیرنده را پایدار نمی‌کند. تماس بعدی دوباره در صف انتظار قرار می‌گیرد.
- **همیشه قبول** تماس‌گیرنده را به فهرست مجاز اضافه می‌کند (در `${RUNTIME_DIR}/.bale-vpn_config.json` زیر کلید `admission` با mode 0600 ذخیره می‌شود). تماس‌های بعدی به‌صورت خودکار پذیرفته می‌شوند.
- **رد** یک `DiscardCall` می‌فرستد تا تونل تماس‌گیرنده بلافاصله قطع شود **و** شناسهٔ تماس‌گیرنده به **فهرست مسدود** اضافه می‌شود — تماس‌های بعدی از این شناسه بی‌صدا رد می‌شوند (بدون اعلان و بدون درخواست انتظار). برای لغو، از فهرست **تماس‌گیرنده‌های مسدود** دکمهٔ Unblock همان ردیف را بزنید.
- در صورت بی‌پاسخ ماندن، درخواست‌های در انتظار بعد از ۶۰ ثانیه به‌طور خودکار رد می‌شوند. این ۶۰ ثانیه تماس‌گیرنده را مسدود نمی‌کند — فقط دکمهٔ صریح Reject این کار را می‌کند.

فهرست مجاز و فهرست مسدود دو طرف منحصربه‌فرد هستند — هر Allow / Reject صریح، تماس‌گیرنده را بین این دو جابه‌جا می‌کند.

تنظیم **حداکثر کلاینت** (۱ تا ۲۵۳، پیش‌فرض ۵) سقف تماس‌های هم‌زمان را تعیین می‌کند. تماس‌های ورودی بیش از این سقف بدون مسدود کردن حذف می‌شوند؛ تماس‌گیرنده می‌تواند بعداً وقتی جا باز شد دوباره تلاش کند.

## احراز هویت

دو راه برای وارد کردن JWT `access_token`:

۱. **OTP موبایل از UI** — شماره را وارد کنید، کد پیامکی را تایپ کنید؛ باینری کوکی را از مسیر استاندارد `web.bale.ai/set-cookie/?jwt=…` می‌گیرد و در `${RUNTIME_DIR}/.bale-vpn_config.json` (mode 0600) ذخیره می‌کند. این روش توصیه‌شده است.

۲. **چسباندن مستقیم توکن** — کوکی `access_token` را از یک نشست لاگین‌شدهٔ `web.bale.ai` در Chrome (DevTools → Application → Cookies) کپی کنید و در تکست‌اریا روی UI بچسبانید.

کد بستن WebSocket `4401` یعنی توکن منقضی شده؛ دوباره وارد شوید.

## امنیت و رمزنگاری

اتصال داده‌ای بین کلاینت و سرور با DTLS رمزنگاری می‌شود، پس ترافیک از دید ناظرهای منفعل شبکه پنهان است. **اما** سرورهای LiveKit بله نقش SFU را بازی می‌کنند و به محتوای رمزگشایی‌شده دسترسی دارند — می‌توانند مقصدها و هر بارِ اپلیکیشنی غیررمزنگاری‌شده را ببینند. در سطح برنامه از TLS (HTTPS، DNS رمزنگاری‌شده و …) استفاده کنید و این تونل را مثل VPN شرکتی‌ای ببینید که گرداننده‌اش را کاملاً نمی‌شناسید.
